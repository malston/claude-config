```markdown
# tdd-pr

Implement a GitHub issue using TDD, review the result, hand off to a PR workflow.

## Arguments

Parse the user's invocation for these arguments:

- `gh_issue` (required) -- GitHub issue number (e.g., `42`) or URL (e.g., `https://github.com/org/repo/issues/42`)
- `branch` (optional) -- branch name to create or checkout
- `--auto` (optional flag) -- skip the review pause and auto-invoke the hand-off skill

If `gh_issue` is missing, ask the user for it before proceeding.

## Prerequisites

Before starting, verify these are available:

1. `gh` CLI is installed and authenticated: run `gh auth status`
2. The working directory is a git repository with a clean working tree
3. If the working tree is dirty, stop and ask the user how to handle uncommitted changes

If any prerequisite fails, stop and tell the user what's missing.
```

````markdown
## Phase 1: Setup

1. **Fetch the issue:**

   ```bash
   gh issue view <gh_issue> --json title,body,labels,assignees
   ```
````

If the issue number is invalid or the repo has no issues enabled, stop and tell the user.

2. **Parse acceptance criteria:**

   Read the issue title and body. Extract a numbered list of acceptance criteria -- concrete, testable behaviors the implementation must satisfy.

   If the issue has no clear acceptance criteria (vague description, no concrete behaviors), stop and present what you found. Ask the user to clarify what "done" looks like before proceeding.

3. **Create or checkout the branch:**
   - If `branch` argument was provided: `git checkout -b <branch>` (or `git checkout <branch>` if it exists)
   - If omitted: generate a branch name from the issue like `issue-<number>-<slugified-title>` (lowercase, hyphens, max 50 chars). Run `git checkout -b <generated-name>`.
   - If the branch already has commits, continue from the current state. Do not reset or rebase.

4. **Present the plan:**

   Show the user:
   - Issue title and link
   - Branch name
   - Numbered acceptance criteria list
   - "Starting TDD loop. I'll implement each criterion using RED-GREEN-REFACTOR."


Append after the Phase 1 section:

```markdown
## Phase 2: TDD Loop

Invoke `superpowers:test-driven-development` to guide each criterion through RED-GREEN-REFACTOR.

For each acceptance criterion from Phase 1:

1. **RED:** Write a failing test that demonstrates the criterion's desired behavior. The test name should reference the criterion.
2. **Verify RED:** Run the test suite. Confirm the new test fails for the expected reason (not a syntax error or import failure).
3. **GREEN:** Write the minimal production code to make the test pass. Do not implement anything beyond what this criterion requires.
4. **Verify GREEN:** Run the full test suite. All tests must pass with pristine output (no warnings, no unexpected stderr).
5. **REFACTOR:** Clean up the implementation -- remove duplication, improve names, simplify logic. Run the full test suite again to confirm everything stays green.

Repeat for the next criterion until all are implemented.

### Failure mode

If the same criterion fails 3 RED-to-GREEN cycles (you write the test but cannot make it pass after 3 implementation attempts), stop the loop. Present to the user:

- Which criterion is stuck
- What was attempted in each cycle
- What error or failure persists
- Ask for guidance before continuing

### Completion

When all criteria have passing tests, announce:

- Number of acceptance criteria implemented
- Total tests added
- "All tests green. Moving to review."
```

### Task 4: Write Phase 3 (Review Checkpoint)

**Files:**

- Modify: `~/.claudeup/ext/skills/tdd-pr/SKILL.md`

- [ ] **Step 1: Append the Phase 3 section to SKILL.md**

Append after the Phase 2 section:

````markdown
## Phase 3: Review Checkpoint

Skip this entire phase if `--auto` flag was provided. Jump to Phase 4.

1. **Commit all work:**

   Stage and commit all changes with a WIP message:

   ```bash
   git add -A
   git commit -m "wip: implement issue #<number> - <issue title>"
   ```
````

2. **Run code review:**

   Spawn a `superpowers:code-reviewer` agent to review the diff between the current branch and the base branch (usually `main`):
   - The reviewer should focus on: correctness, test quality, missed edge cases, style consistency
   - Capture the reviewer's findings

3. **Present the summary:**

   Show the user:
   - **Acceptance criteria status:** checklist mapping each criterion to its test(s)
   - **Test results:** pass count, any coverage info if available
   - **Code review findings:** issues and suggestions from the reviewer
   - **Diff stats:** files changed, lines added/removed

4. **Wait for user input:**
   - If the user says "proceed", "lgtm", "looks good", or similar: move to Phase 4
   - If the user gives specific feedback: address it, re-run the relevant tests, re-run the code reviewer, and re-present the summary
   - If the user says "stop": leave the work on the branch, tell them the branch name and how to resume later. Do not invoke Phase 4.


````markdown
## Phase 4: Hand-off

1. **Compute diff stats:**

   ```bash
   git diff --stat main..HEAD
   ```
````

Count: number of files changed, total lines added + removed.

2. **Select the hand-off skill:**
   - If the diff touches **3 or fewer files** AND **100 or fewer total lines changed**: recommend `/commit-push-pr`
   - Otherwise: recommend `/git-workflow`

3. **Present the recommendation:**

   If `--auto` flag was set: invoke the recommended skill immediately without asking.

   Otherwise, tell the user:
   - Which skill is recommended and why (diff size)
   - "Say 'proceed' to use the recommended skill, or name a different one (e.g., '/commit-push-pr' or '/git-workflow')."

4. **Invoke the chosen skill.**

   The hand-off skill takes over from here. tdd-pr's job is done.

